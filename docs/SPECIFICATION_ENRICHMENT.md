# Metagomics 2 — Single-Sample GO × Taxonomy Enrichment Analysis

## 1. Purpose and Motivation

In metaproteomics, the Metagomics 2 pipeline assigns each peptide a taxonomy (via LCA) and a set of Gene Ontology (GO) terms (via closure union). The GO × taxonomy cross-tabulation (`go_taxonomy_combo.csv`) reports the abundance of every observed (taxon, GO term) pair. However, raw abundance alone does not tell the user whether a specific GO term is **unusually concentrated** in a specific taxon compared to the sample-wide background.

This enrichment analysis answers two complementary questions for each (taxon, GO term) pair:

1. **Is GO term `g` enriched or depleted within taxon `t`?** — tested by taxonomy shuffle (`pvalue_go_for_taxon`)
2. **Is taxon `t` enriched or depleted for GO term `g`?** — tested by GO shuffle (`pvalue_taxon_for_go`)

Because DIA peptide abundances are continuous (often very large floating-point values), standard count-based tests like Fisher's exact test or chi-square are inappropriate. Instead, significance is estimated using **empirical null distributions** generated by Monte Carlo randomization (permutation testing).

---

## 2. Scope and Limitations

This analysis applies to a **single sample**. It is a within-sample enrichment test, not a between-sample differential test.

It conditions on the observed peptide-level assignments and does **not** model uncertainty in:

- Peptide identification
- Protein inference
- GO annotation correctness
- Taxonomic assignment correctness
- Biological replication

---

## 3. Input Data

For each annotated peptide in a single sample, the enrichment module requires:

| Field | Type | Source |
|-------|------|--------|
| `peptide` | string | Peptide sequence identifier |
| `quantity` | float | DIA abundance (continuous, non-negative) |
| `taxonomy_nodes` | set of int | Full taxonomic closure from LCA to root |
| `go_terms` | set of string | Full GO closure (union of all ancestor terms) |
| `is_annotated` | bool | Whether the peptide has any annotations |

These come from `PeptideAnnotation` objects produced by the annotation pipeline (`core/annotation.py`).

### Peptide pool filtering

Before any enrichment calculation, peptides are filtered to those with **both** non-empty `taxonomy_nodes` and non-empty `go_terms`. Unannotated peptides and peptides with only taxonomy or only GO annotations are excluded. This means the enrichment denominators are based on the **doubly-annotated peptide pool**, not total sample abundance.

---

## 4. Observed Quantities

For a peptide pool of `P` annotated peptides, define:

| Symbol | Definition |
|--------|-----------|
| `A_total` | Sum of abundances across all peptides in the pool |
| `A_TAX(t)` | Sum of abundances of peptides whose taxonomy closure contains taxon `t` |
| `A_GO(g)` | Sum of abundances of peptides whose GO closure contains GO term `g` |
| `A_JOINT(t, g)` | Sum of abundances of peptides with both taxon `t` and GO term `g` |

### Derived proportions

| Symbol | Formula | Interpretation |
|--------|---------|----------------|
| `p_obs(t, g)` | `A_JOINT(t, g) / A_TAX(t)` | Fraction of taxon `t`'s abundance with GO term `g` |
| `p_bg(g)` | `A_GO(g) / A_total` | Sample-wide background proportion of GO term `g` |

### Test statistic: log2 enrichment

```
L_obs(t, g) = log2( (p_obs(t, g) + eps) / (p_bg(g) + eps) )
```

Where `eps = 1e-10` is a small constant for numerical stability.

- `L_obs > 0` → GO term `g` is enriched within taxon `t`
- `L_obs < 0` → GO term `g` is depleted within taxon `t`
- `L_obs ≈ 0` → no enrichment or depletion

---

## 5. Randomization Modes

Two complementary randomization modes are implemented. Both are **aspect-stratified** to control for cross-aspect annotation-quality confounds (see Section 6).

### 5.1 Taxonomy Shuffle Mode (`pvalue_go_for_taxon`)

**Question:** Is GO term `g` unusually enriched or depleted within taxon `t`, compared to what would be expected if taxonomy assignments were unrelated to GO assignments?

**What stays fixed:** peptide abundances, peptide GO closures

**What is shuffled:** peptide taxonomy closure objects (the entire lineage from LCA to root is shuffled as an atomic unit)

**Aspect stratification:** For each GO aspect (molecular_function, biological_process, cellular_component), the shuffle pool is **restricted to peptides that have at least one GO term in that aspect**. Tested pairs involving GO terms in that aspect are evaluated only against this restricted pool. This ensures the background proportion `p_bg` is computed relative to peptides with the relevant annotation type, preventing root GO terms from appearing spuriously enriched due to cross-aspect annotation-quality differences. See Section 6 for the detailed motivation.

**Null hypothesis:** Conditional on the observed peptide abundances and GO closures, and within the aspect-restricted peptide pool, taxonomy assignments are exchangeable across peptides.

### 5.2 GO Shuffle Mode (`pvalue_taxon_for_go`)

**Question:** Is taxon `t` unusually enriched or depleted for GO term `g`, compared to what would be expected if GO assignments were unrelated to taxonomy assignments?

**What stays fixed:** peptide abundances, peptide taxonomy closures

**What is shuffled:** peptide GO closures, **independently per aspect**

**Aspect stratification:** For each iteration, three independent permutations are applied — one for molecular_function GO terms, one for biological_process, one for cellular_component. Each peptide's shuffled GO closure is the union of its randomly drawn MF closure from one peptide, BP closure from another, and CC closure from a third. This breaks the correlation between "has MF annotation" and "has BP annotation" that exists when well-annotated peptides have comprehensive coverage across all three aspects.

**Null hypothesis:** Conditional on the observed peptide abundances and taxonomy assignments, GO assignments within each aspect are independently exchangeable across peptides.

---

## 6. Aspect Stratification — Motivation

### The annotation-quality confound

In real metaproteomics data, annotation quality varies systematically across taxa. For example:

- **Bacterial peptides** often match well-characterized SwissProt proteins with comprehensive GO annotations (MF + BP + CC terms)
- **Eukaryotic peptides** may match less-characterized proteins with partial GO annotations (e.g., only BP terms)

Without aspect stratification, this creates two confounds:

**Confound 1 (taxonomy shuffle):** When testing whether `biological_process` root is enriched in Bacteria, the shuffle pool includes peptides that have GO annotations but no BP terms at all (e.g., MF-only peptides). These dilute the background proportion `p_bg`, making Bacteria's near-universal BP coverage look enriched. The fix: restrict the taxonomy shuffle pool to peptides that have at least one BP term. Among these, the BP root is nearly universal, so Bacteria's 99.97% coverage is unremarkable.

**Confound 2 (GO shuffle):** When shuffling entire GO closures as atomic units, well-annotated closures (MF + BP + CC) move together. The null preserves the correlation that "has MF" implies "has BP." The fix: shuffle MF, BP, and CC closures independently. This tests whether a specific MF term is enriched in a taxon, independent of that taxon's BP/CC annotation quality.

### Three GO aspects

The Gene Ontology has three independent root namespaces:

| Aspect | Root GO ID | Root Name |
|--------|-----------|-----------|
| Molecular Function | GO:0003674 | molecular_function |
| Biological Process | GO:0008150 | biological_process |
| Cellular Component | GO:0005575 | cellular_component |

Every GO term belongs to exactly one aspect (determined by its `namespace` field in the OBO file). The aspect of each GO term is provided to the enrichment module via the `go_namespaces` mapping, which is built from the GODAG in the pipeline runner.

## 7. Monte Carlo Procedure

Let `B` be the number of iterations (default: 1000, configurable up to 10000).

### 7.1 For each shuffle mode:

1. Compute observed `L_obs(t, g)` for every tested (taxon, GO) pair
2. For `b = 1, ..., B`:
   a. Apply the appropriate permutation (see Section 5)
   b. Recompute joint abundances, marginal abundances, and `L_null_b(t, g)` for all tested pairs
3. Compute the null mean: `mean_null(t, g) = mean(L_null_1, ..., L_null_B)`
4. Compute two-sided p-value:

```
count_extreme = |{ b : |L_null_b - mean_null| >= |L_obs - mean_null| }|
p_value = (1 + count_extreme) / (B + 1)
```

The `+1` in numerator and denominator ensures p-values are never exactly zero and are uniformly distributed under the null.

### 7.2 Multiple testing correction

Benjamini-Hochberg (BH) FDR correction is applied to the raw p-values across all tested pairs within each mode:

1. Sort p-values in ascending order
2. For rank `i` (1-indexed): `adjusted_i = min(p_i * n / i, 1.0)`
3. Enforce monotonicity: backward cumulative minimum so `q_i <= q_{i+1}`
4. Restore original order

The resulting q-values represent the expected false discovery rate if all pairs with q-value ≤ threshold are declared significant.

### 8.3 Internal data representation (`_IndexMaps`)

To avoid Python-level loops during Monte Carlo iterations, all peptide-level data is pre-converted to integer-indexed NumPy arrays:

| Field | Shape | Description |
|-------|-------|-------------|
| `abundance` | `(P,)` | Peptide abundance vector |
| `tax_mem` | `(P, T)` | Boolean taxonomy membership matrix |
| `go_mem` | `(P, G)` | Boolean GO membership matrix |
| `pair_tax_idx` | `(K,)` | Tax column index for each of the K tested pairs |
| `pair_go_idx` | `(K,)` | GO column index for each of the K tested pairs |
| `obs_log2` | `(K,)` | Observed log2 enrichment for each tested pair |
| `go_aspect_masks` | list of `(G,)` bool | Per-aspect column masks for GO matrix (one per namespace) |

### 8.4 Vectorized shuffle iteration

Each Monte Carlo iteration performs these steps (all NumPy, no Python per-pair loops):

```python
perm = rng.permutation(n_peptides)
shuffled = shuffle_mem[perm]                      # row permutation (atomic closure shuffle)
weighted_shuffled = shuffled * abundance[:, None]  # abundance-weighted membership
joint_matrix = weighted_shuffled.T @ fixed_mem     # (T, G) joint abundance matrix
a_shuffled = shuffled.T @ abundance                # marginal abundance of shuffled side

# Extract values for all K tested pairs at once via fancy indexing
joint_vals = joint_matrix[pair_tax, pair_go]       # (K,)
denom_vals = a_shuffled[pair_tax]                  # (K,)
bg_vals = a_fixed[pair_go]                         # (K,)

p_obs_null = (joint_vals + eps) / (denom_vals + eps)
p_bg = (bg_vals + eps) / (a_total + eps)
null_values[b] = np.log2(p_obs_null / p_bg)        # (K,) null statistic
```

The matrix operation `@` uses BLAS internally and is inherently multi-threaded.

## 9. Output Columns

The enrichment analysis adds four columns to `go_taxonomy_combo.csv`. These columns are empty strings when enrichment is not computed (backward-compatible).

| Column | Type | Description |
|--------|------|-------------|
| `pvalue_go_for_taxon` | float | Two-sided empirical p-value from **taxonomy shuffle**. Tests: "Is GO term `g` enriched or depleted within taxon `t`?" Low values indicate the observed concentration of this GO term in this taxon is more extreme than expected under random taxonomy assignment (within the aspect-restricted peptide pool). |
| `pvalue_taxon_for_go` | float | Two-sided empirical p-value from **GO shuffle** (aspect-stratified). Tests: "Is taxon `t` enriched or depleted for GO term `g`?" Low values indicate this taxon's share of this GO term's abundance is more extreme than expected under random GO assignment (with independent shuffling per GO aspect). |
| `qvalue_go_for_taxon` | float | Benjamini-Hochberg adjusted `pvalue_go_for_taxon`. Controls the false discovery rate across all tested pairs. |
| `qvalue_taxon_for_go` | float | Benjamini-Hochberg adjusted `pvalue_taxon_for_go`. Controls the false discovery rate across all tested pairs. |

### Interpreting p-values vs q-values

- **p-value**: The probability of observing enrichment as extreme as the actual data under the null model. Does not account for multiple testing.
- **q-value**: The expected false discovery rate if all pairs with q-value ≤ this threshold are declared significant. Accounts for the thousands of (taxon, GO) pairs being tested simultaneously.
- **Two-sided**: Both enrichment (GO term more concentrated in the taxon than expected) and depletion (GO term less concentrated than expected) contribute to significance. A low p-value does not by itself indicate the direction — check the `fraction_of_taxon` and `fraction_of_go` columns for that.

### Which column to use?

- Use `qvalue_go_for_taxon` to find GO terms that are unusually concentrated in specific taxa
- Use `qvalue_taxon_for_go` to find taxa that are unusually concentrated for specific GO terms
- The two modes ask closely related but not identical questions and may produce different results for the same (taxon, GO) pair

## 12. Performance Characteristics

- **Single-process execution**: All Monte Carlo iterations run in a single process. An earlier multiprocessing implementation using `multiprocessing.Pool` with `fork` mode caused production failures due to inherited signal handlers and SQLite connection corruption in Docker. It was removed in favor of single-process NumPy-vectorized execution.
- **BLAS threading**: The matrix operations (`@`) use NumPy's internal BLAS implementation, which is inherently multi-threaded. This provides parallelism without the risks of Python multiprocessing.
- **Fully vectorized**: There are no Python-level per-pair loops in the hot path. All per-pair operations use NumPy fancy indexing.
- **Memory**: Stores a `(B, K)` array of null statistics where `B` = iterations and `K` = number of tested pairs. For 1000 iterations and 100,000 pairs, this is ~800 MB. For very large datasets, a streaming approach (running counters) could replace this, but is not currently implemented.
- **Aspect stratification overhead**: The taxonomy shuffle runs 3 separate shuffles (one per GO aspect), each on a smaller peptide pool. The GO shuffle runs 3 permutations per iteration instead of 1. Total work is roughly 3× a non-stratified approach, but each individual shuffle operates on a smaller matrix.

## 14. Design Decisions and Trade-offs

### Why Monte Carlo randomization instead of analytical tests?

DIA abundances are continuous and often span many orders of magnitude. Standard count-based tests (Fisher's exact, chi-square) assume discrete data and are inappropriate. Monte Carlo randomization is distribution-free and works directly with continuous abundances.

### Why two shuffle modes?

The two modes ask subtly different questions. Taxonomy shuffle tests "is this GO term concentrated in this taxon?" while GO shuffle tests "is this taxon concentrated for this GO term?" With propagated closures (where each peptide contributes to many taxa and many GO terms), these can produce meaningfully different results.

### Why aspect-stratified shuffling?

Without stratification, annotation-quality differences across taxa create pervasive confounds. Well-annotated taxa (e.g., Bacteria in SwissProt) appear enriched for root-level GO terms simply because they have more complete annotation. Aspect stratification isolates the MF, BP, and CC questions from each other, testing whether a *specific* GO term is enriched rather than whether overall annotation quality differs.

### Why single-process instead of multiprocessing?

An initial implementation used `multiprocessing.Pool` with `fork` mode. In production (Docker), forked child processes inherited the parent worker's SIGTERM signal handlers and SQLite connections, causing the worker process to terminate during enrichment computation. The fix was to remove multiprocessing entirely and rely on NumPy's internal BLAS threading for parallelism. The fully vectorized implementation is fast enough for typical dataset sizes.

### Why store all null values instead of running counters?

The two-sided p-value formula requires comparing `|L_null - mean_null|` against `|L_obs - mean_null|`. Since `mean_null` is only known after all iterations complete, we store the full `(B, K)` matrix of null statistics. For the supported range of B ≤ 10000 and typical K (tested pairs), this fits comfortably in memory. A streaming approach using Welford's online algorithm could be implemented if memory becomes a concern.

### Why `(1 + count_extreme) / (B + 1)` instead of `count_extreme / B`?

The `+1` correction (Davison and Hinkley, 1997) ensures p-values are never exactly zero and are uniformly distributed under the null. Without it, p-values of exactly 0 are possible when no null iteration is as extreme as the observation, which can cause issues with downstream FDR correction.

## 15. Future Extensions

- **Feature filtering**: Exclude very high-level GO terms (e.g., within N levels of root) from testing to reduce the multiple-testing burden and focus on specific terms
- **Streaming null statistics**: Replace the `(B, K)` null matrix with running Welford counters for memory efficiency on very large datasets
- **Confidence-weighted randomization**: Weight shuffles by peptide identification confidence
- **Parallel execution**: If multiprocessing is re-introduced, use `spawn` mode instead of `fork` to avoid inheriting parent process state, or use thread-based parallelism
- **Directional p-values**: Report separate enrichment and depletion p-values in addition to the two-sided p-value
- **Diagnostic output**: Optionally report null mean, null standard deviation, and null quantiles for quality control
